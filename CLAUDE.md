```bash
uv run pytest                                          # Run all tests
uv run pytest -m "not slow"                            # Skip slow tests
uv run pytest tests/unit/test_foo.py                   # Run a single test file
uv run pytest -m "not slow" --cov=kibad_llm --cov-report=term-missing  # With coverage
```

```bash
uv run -m kibad_llm.predict    # Extract structured data from PDFs (Hydra-configured)
uv run -m kibad_llm.evaluate   # Evaluate predictions against reference data
```

## Architecture

KIBAD-LLM extracts structured information from scientific PDFs using LLMs, targeting ecosystem study features. The pipeline has two main stages: **prediction** and **evaluation**.

### Prediction pipeline

```
PDFs → pymupdf4llm (→ markdown) → Extractor + LLM + Pydantic schema → predictions.jsonl
```

- Entry point: `src/kibad_llm/predict.py` (Hydra app)
- Extractors in `src/kibad_llm/extractors/` implement different chunking/retry strategies (simple, chunking, union, repeat, conditional). Each extractor takes markdown text and a schema, calls an LLM, and returns structured output.
- LLM backends in `src/kibad_llm/llms/` wrap LlamaIndex: `openai.py`, `vllm_in_process.py`, `openai_like.py`
- Schemas in `src/kibad_llm/schema/` define Pydantic models for the ecosystem features being extracted

### Evaluation pipeline

```
predictions.jsonl + reference dataset → metrics (F1, confusion matrix, etc.)
```

- Entry point: `src/kibad_llm/evaluate.py` (Hydra app)
- Metrics in `src/kibad_llm/metrics/`
- Datasets in `src/kibad_llm/dataset/` (loads predictions, reference CSVs, JSON)

### Configuration

All configuration uses **Hydra** — base configs in `configs/`, with structured overrides:

- `configs/predict.yaml` / `configs/evaluate.yaml` — top-level configs
- `configs/extractor/` — extractor type + LLM + schema combos
- `configs/experiment/` — reproducible experiment setups (multirun-ready)

### Data integration

`src/kibad_llm/data_integration/` contains one-off scripts for downloading from Zotero, syncing with Nextcloud, and converting from a PostgreSQL Faktencheck DB (`podman/faktencheck-db/`).

### Key supporting modules

- `src/kibad_llm/config.py` — project paths and env var loading
- `src/kibad_llm/utils/` — logging (loguru), path helpers, job return handling
- `src/kibad_llm/hydra_callbacks/` — custom Hydra callbacks for output management

## Environment

Copy `.env.example` to `.env` and fill in:

- `OPENAI_API_KEY` — for OpenAI models
- `HF_TOKEN` — for gated Hugging Face models (Gemma3, etc.)
- `DB_USER` / `DB_PASSWORD` — for the Faktencheck PostgreSQL DB (optional)

## Docs

Properdocs (based on mcdocs) site lives in `docs/`; deployed to GitHub Pages. Build locally with `uv run properdocs serve`.

Since the docs are autogenerated from the docstrings, **all docstrings must be valid markdown**. Our documentation is generally focussed on producing a very good static page through properdocs.
